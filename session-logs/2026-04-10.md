### 2. gl_flange.png — two-pass bore interior fix

Problem: bore interior of `llm_steel_pipe_flange_od_bore.stl` rendered white/bright. The trimesh GL renderer illuminates bore interior surfaces with ambient light — they appear near-white (R,G,B all > 230). Standard corner-based flood-fill cannot reach the bore interior because it is geometrically enclosed.

Root cause: GL ambient lighting + bore geometry = no path from image corners to bore pixels.

Fix — two-pass approach:
1. Flood-fill from all 4 image corners (threshold=30) to identify true background pixels
2. Replace all remaining near-white pixels (R > 230 AND G > 230 AND B > 230) with dark background color `#0d1117`

Aluminum geometry maxes out at ~210 brightness, so the 230 threshold safely separates bore interior (white, > 230) from aluminum faces (steel-blue, ≤ 210).

Script: `.scratch/fix_impeller_flange.py`

## Key technical finding

**trimesh GL bore interior whiteness**: bore/through-hole interiors appear white/bright due to ambient lighting on interior faces. Cannot be fixed with corner-based flood-fill alone. Two-pass fix: (1) flood-fill from corners at thresh=30, (2) threshold all remaining near-white pixels (>230 all channels) to background. Works because aluminum geometry maxes at ~210 brightness, creating a clean separation.

# Session Log — 2026-04-10 (Part 5: Viewer + Visual Verifier + Gallery Scripts)

## What was built/fixed

### 1. viewer.html — black screen fix for large STL models

Three.js default `camera.far=1000` clips terrain and civil-scale geometry, producing a black screen.

Fix: after STL load, compute model bounding sphere radius, then:
- `camera.far = radius * 10`
- fog end = `radius * 8`
- `camera.updateProjectionMatrix()`
- `orbitControls.maxDistance = radius * 5`

Result: any STL at any scale renders without clipping.

### 2. viewer.html — terrain Z-up auto-rotation

Terrain STLs are generated in Z-up orientation. Detection condition:
`y > 15*z AND y > 1000 AND x > 200` (metres-scale bounds).

When triggered: rotate the scene -90° around the X axis so Z becomes screen-up and terrain lies flat on screen.
No user action required — the viewer auto-corrects on load.

### 3. visual_verifier.py — 4 improvements

**4th isometric view**: Added isometric projection using Rx@Ry rotation matrix (Rx = -35.264° about X, Ry = 45° about Z). Provides a standard engineering isometric perspective the vision model can use to assess overall 3D shape.

**Actual mm dimensions in view titles**: Titles now read "Top view — 213 x 213 mm" using `mesh.bounding_box.extents`. Vision model uses these to cross-check against spec without inferring scale.

**Volume plausibility + Euler hole count**:
- Volume check: actual volume should be 5–90% of bbox volume (below 5% = shell, above 90% = solid brick, feature generation failed)
- Euler characteristic: `holes = 1 - (V - E + F) / 2` gives topological handle count; compared against expected `n_bolts + 1` (bore) for flanges and similar parts

**Winding consistency check**: Samples 200 face pairs sharing an edge; checks opposing half-edge orientation. Score < 160/200 = inconsistent winding warning. Non-manifold meshes always fail.

**Cross-validation threshold lowered from 0.90 to 0.80**: Now cross-validates any primary provider (Groq/Gemini) at confidence ≥ 0.80. Catches more borderline false passes.

**Spec-driven proportion expectations in vision prompt**: Adds od/height ratio, bore/od ratio, blade count, fin count as explicit expectations. Reduces hallucinated PASS on obviously wrong proportions.

**FAIL bias instruction**: "When in doubt, report FAIL. A false FAIL is recoverable; a false PASS wastes CAM time and material."
