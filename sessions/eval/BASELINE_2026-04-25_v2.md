## Root cause: token budget, not request budget

Our system prompt for `make_plan` adds up to:

  - ENGINEERING_PRACTICE_PROMPT       ~5,000 chars
  - _OPS_SCHEMA (op vocabulary)       ~6,000 chars
  - _FEW_SHOT_EXAMPLE                 ~10,000 chars
  - W2 retrieved API docs (top-5)     ~2,000 chars
  - W2 retrieved few-shots (top-2)    ~3,000 chars

Total = ~26,000 chars ≈ 6,500 tokens per request.

Groq's free tier on `llama-3.3-70b-versatile` is **12,000 TPM**.
That means at 6,500 tokens per request we can fit ~1.8 requests
per minute before TPM saturation, regardless of how much pacing
we add for RPM.

Math: 50 prompts × 6,500 tokens ÷ 12,000 TPM ≈ 27 minutes minimum.

## Two paths to a real 50-prompt run

### Path 1 (cheap, slow): pace at 35 sec/request
35s pacing keeps us under TPM. 50 prompts in ~30 minutes. Can run
unattended.

### Path 2 (proper, fast): trim system prompt for eval mode
Cut the engineering prompt to top-5 relevant tables, drop the
retrieved API blocks (the few-shots already cover the ops we need
for the eval set), and leave just the op schema + 2-3 few-shots.
Target: ≤2,500 tokens per request. Eval finishes in ~5 minutes.

Both paths are mechanical changes to existing code; this is not
blocking on credits anymore.
